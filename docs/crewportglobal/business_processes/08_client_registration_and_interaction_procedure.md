## 1. Purpose

This document defines the operational procedure for registering and interacting with CrewPortGlobal users, seafarers, employer-side clients, internal specialists and reviewers.

It also records the required site/menu transition:

```text
Public pages provide general information and a clear login/registration entry.
Functional pages are generated after registration/authentication according to group membership, confirmed cards, relationships and authority evidence.
```

The goal is to stop mixing public information pages with functional working pages.

## 2. Core Public Site Rule

The public website should become an information and acquisition surface.

Public pages should explain:

```text
what CrewPortGlobal does

employer-side demand and B2B service model

```

Public pages should not expose operational workspaces.

The public menu should emphasize:

```text

Trust and safety
Login / Registration
Language
```

The primary functional entry should be:

```text
Login / Registration
```

After login, the personal cabinet and internal service menu are generated by the backend from confirmed access context.

## 4. Route Transition Principle

Existing public/action routes may remain temporarily during migration, but their future role changes.

| Current route | Future role |
|---|---|
| `/register/` | Public login/registration entry and route to `/me/` after authentication |
| `/create-profile/` | Transitional route; future seafarer work moves to `/me/seafarer/` |
| `/post-vacancy/` | Transitional route; future employer demand work moves to `/me/employer/` and `/me/crew-requests/` |
| `/vacancies/` | Public or limited public information only; personalized matching after login |
| `/verify/` | Transitional internal operator route; future review work moves to `/me/reviews/` or team pages |
| `/team/` | Protected team entry; future team work links generated by group membership |
| `/admin/access/` | Protected Project Owner/admin access and audit |

No route transition should weaken authentication or authorization.

## 5. Registration Procedure Overview

Every new person starts with:

```text
Physical Person Registration Card
Service Account / Login Card
```

The ordinary rule is:

```text
physical person self-registers
```

An agent, employer, seafarer, shipowner or internal specialist does not become a platform actor through another participant's account. A third party may invite the person or prepare a draft context for review, but the physical person must personally register or claim the account before ordinary participant rights are activated.

Then the authenticated person selects or receives one or more platform capacities:

```text
seafarer workforce / supply
shipowner / employer-side requester / demand
agent organization representative
internal team member
support case participant
review/control assignment
Project Owner/admin assignment
```

Each capacity creates draft or confirmed cards, subject to its own authority evidence and review status.

The personal cabinet assembles visible work areas from those cards.

Agent appointment is optional. A registered participant may act personally, appoint an agent representative, replace an agent representative or revoke representation at any stage allowed by the relevant workflow and contract-critical blockers.

## 6. Step 1 - Public Entry

The user arrives at the public website.

Public interface should offer:

```text
general explanation
seafarer value proposition
employer-side value proposition
no-fee statement
trust and review statement
Login / Registration button
```

The user should not need to choose from many functional pages before registration.

The first action is:

```text
Login / Registration
```

## 7. Step 2 - Physical Person Registration

The system creates or confirms a Physical Person Registration Card.

Minimum process:

1. collect display name;
2. collect e-mail;
3. optionally collect phone and preferred language;
4. record source channel;
5. request consent where required;
6. perform duplicate check;
7. create draft/active physical person state.

Required result:

```text
physical_person_id
registration_state
duplicate_check_status
consent_status
```

The system must not treat this as authority to act for a company or vessel.

## 8. Step 3 - Authentication

The system creates or confirms Service Account / Login Card and authenticates the user.

Allowed first implementation:

```text
e-mail code authentication
```

Authentication result:

```text
active service account
active session
linked physical person
```

Authentication does not grant broad data visibility.

## 9. Step 4 - Intended Path Selection

After authentication, the user selects or receives an intended path.

Options:

```text
I am a seafarer
I am a shipowner / I request crew / I represent an employer-side client
I represent an agent organization
I was invited by the CrewPortGlobal team
I need support
```

The selected path creates a draft card and tasks.

It does not create final authorization.

Final authorization depends on the capacity:

1. seafarer capacity requires the user's own seafarer profile context;
2. shipowner/employer company capacity requires company authority evidence unless the person is an individual owner where legally applicable;
3. agent capacity requires agent organization membership and authority review;
4. internal team capacity requires admin approval;
5. representative actions require a separate appointment, verified authority or approved enhanced-authority exception.

## 10. Seafarer Registration Procedure

The seafarer flow creates supply-side data.

Procedure:

1. create Seafarer Workforce Card;
2. collect rank;
3. collect department;
4. collect availability date/status;
5. collect expected compensation and accepted terms;
6. collect accepted vessel types;
7. collect contract duration preference;
8. collect documents/certificates metadata;
9. collect experience summary;
10. show no-recruitment-fee acknowledgement;
11. save draft or submit for review;
12. create review/correction tasks as needed;
13. show seafarer workspace in personal cabinet.

Primary output:

```text
seafarer supply card ready for review and future matching
```

Forbidden:

```text
mandatory recruitment fee
paid service as condition for job access
automatic approval without human review where review is required
candidate presentation without reviewed matching result
```

## 11. Employer-Side Registration Procedure

The employer-side flow creates demand-side requester, company and request data.

Procedure:

1. create Employer-Side Requester Card;
2. collect requester type;
3. collect business e-mail/contact;
4. create or link Company Context Card;
5. collect company context;
6. request Authority Evidence Card;
7. create or link Vessel Context Card where required;
8. create Vacancy / Crew Request Card draft;
9. collect demand fields;
10. submit authority/request for review;
11. create review/correction tasks as needed;
12. show employer-side workspace in personal cabinet.

Primary output:

```text
employer-side demand card ready for authority review and future matching
```

Forbidden:

```text
active request without authority review
request visibility beyond company/vessel/request scope
candidate presentation without human-reviewed recommendation
billing activation without approved payment/entitlement rule
```

## 12. Internal Team Invitation Procedure

Internal team members must not use public seafarer/employer registration as the normal staff onboarding mechanism.

Procedure:

1. Project Owner or authorized admin creates/links physical person;
2. service account is authenticated;
3. user is added to approved group;
4. role/permission is inherited through group;
5. visible team pages are generated after login;
6. assigned tasks and client scopes define actual records visible.

Important:

```text
group membership opens service area
task/client/review relationship controls record visibility
```

## 13. Reviewer / Controller Interaction Procedure

Reviewers and controllers work only on assigned or scoped cases.

Procedure:

1. review/control task appears in `Мои задачи`;
2. reviewer opens assigned card or case;
3. system shows only required evidence and fields;
4. reviewer checks conflict-of-interest;
5. reviewer returns correction, approves limited/verified, rejects or escalates;
6. decision creates audit event;
7. next task is created for responsible group or client.

Forbidden:

```text
approve own submission
delete audit history
let AI final-approve
browse all records without control reason
```

## 14. Support Interaction Procedure

Support works through assigned support cases and limited card context.

Procedure:

1. support case or waiting-state task appears;
2. support sees only required context;
3. support sends approved response or requests missing data;
4. support updates communication history;
5. support escalates complaint/compliance/payment issue;
6. support closes only when outcome is recorded.

Support must not approve authority, company, vessel, payment, compliance or candidate presentation decisions.

## 15. Matching And Candidate Recommendation Procedure

Matching starts only when:

```text
employer-side demand exists
authority is reviewed or sufficiently active for limited workflow
seafarer supply exists
minimum matching fields are available
```

Procedure:

1. system compares demand and supply;
2. matching summary is created;
3. AI may prepare draft summary if allowed;
4. human reviewer checks fit and risk;
5. reviewer approves or rejects recommendation;
6. approved recommendation becomes visible to employer-side requester according to entitlement/scope;
7. employer response is recorded;
8. next action is generated.

The platform result is:

```text
reviewed candidate recommendation
```

Not:

```text
automatic hiring
guaranteed placement
automatic legal/employment approval
```

## 16. Interaction Procedure In The Personal Cabinet

After registration/authentication, the user lands in the personal cabinet.

Default screen:

```text
Мои задачи open
all other visible cards collapsed
```

The user works in this order:

1. review `Мои задачи`;
2. open the task that needs action;
3. update the related card/form;
4. save draft or submit for review;
5. close task where allowed;
6. open other cards only when needed;
7. check My clients for follow-up and retention work.

The cabinet should not show a long page of all open cards.

## 17. Functional Page Generation Rule

Functional pages are generated after login based on:

```text
group membership
confirmed cards
card relationships
authority evidence
tasks
review/control assignments
allowed actions
```

Examples:

```text
seafarer workspace appears after Seafarer Workforce Card exists
employer workspace appears after Employer-Side Requester Card exists
company section appears only for visible company contexts
vessel section appears only for visible vessel contexts
crew request section appears only for visible requests
review section appears only for assigned review/control work
admin section appears only for approved admin scope
```

The menu must not show functional pages only because they exist in the website.

## 19. Menu Acceptance Criteria

Future menu implementation should pass:

1. unauthenticated public user sees general information and login/registration;
2. unauthenticated public user does not see private work menus;
3. authenticated seafarer sees seafarer workspace only when card/context exists;
4. authenticated employer-side requester sees only authorized demand-side work areas;
5. internal user sees team areas only by group membership and task/control scope;
6. Project Owner sees admin/audit areas by approved admin scope;
7. functional menu items use access-check output;
8. direct URL access still checks backend authorization;
9. public documents remain reachable where intended;
10. old functional routes either redirect safely to cabinet or remain transitional with the same checks.

## 20. Task And SLA Interaction Rule

Every process step that requires user or team action should create a task.

Task examples:

```text
complete physical person registration
verify e-mail
complete seafarer profile
upload certificate metadata
submit profile for review
submit authority evidence
complete company context
complete crew request
review authority evidence
review crew request
prepare matching summary
approve candidate recommendation
follow up employer response
follow up seafarer correction
```

All active tasks must have:

```text
owner or group
deadline
SLA color
source card
required action
visibility scope
```

## 21. Handoff Procedure

Handoffs must be explicit.

Each handoff records:

```text
from group
to group
from user
to user if known
source card
reason
required next action
deadline
visibility scope
audit reference
```

The responsible manager relationship should remain unless formally reassigned.

## 22. Communication Procedure

Communication with users and clients should be card-linked.

Every meaningful contact should record:

```text
contact date
contact channel
contact direction
summary
next contact date
related card
related task
responsible user
```

Communication should not create authority by itself unless reviewed as evidence under an approved evidence type.

## 23. AI Role In Interaction Procedure

AI may assist with:

```text
drafting messages
summarizing cards
detecting missing fields
preparing matching summary drafts
suggesting next action
highlighting SLA risk
translating routine text where approved
```

AI must not:

```text
approve registration authority
approve employer request authority
approve seafarer readiness
approve candidate presentation
make employment decision
make payment decision
grant access
send final legally binding message without human approval
```

## 25. Transition From Current Site

The current site already contains functional pages and public/document pages.

Transition should happen in phases:

### Phase 1 - Documentation and design freeze

```text
BP-003 through BP-008 define the target model.
No public functional route should be expanded without checking this model.
```

### Phase 2 - Public menu simplification

```text
Move public menu to general information and Login / Registration.
Remove or de-emphasize direct functional links for unauthenticated users.
Keep direct URLs safe during transition.
```

### Phase 3 - Cabinet shell

```text
Create /me/ shell.
Render Мои задачи first and open.
Render other cards collapsed by default.
Use safe mocked or limited backend data first if needed.
```

### Phase 4 - Registration routing

```text
/register/ routes to physical person + service account flow.
/create-profile/ routes authenticated users to /me/seafarer/.
/post-vacancy/ routes authenticated users to /me/employer/ or /me/crew-requests/.
```

### Phase 5 - Access-check enforcement

```text
Every functional API and page uses the access-check contract.
```

## 26. Procedure Acceptance Criteria

Future implementation should pass:

1. public user sees general information and login/registration;
2. public user does not see private work menu;
3. registration creates physical person first;
4. authentication confirms service account only;
5. intended path creates draft card, not final authority;
6. seafarer gets supply-side workspace after card creation;
7. employer-side requester gets demand-side workspace after card creation;
8. company/vessel/request actions require scoped authority;
9. internal team pages appear only by group and assignment;
10. every actionable step creates a task;
11. `Мои задачи` is first and open;
12. other cards are collapsed by default;
13. matching recommendation requires human review;
14. public route transition does not weaken URL-level checks;
15. audit records material steps without secrets.
