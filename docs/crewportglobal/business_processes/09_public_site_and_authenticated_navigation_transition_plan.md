## 1. Purpose

This document defines the transition plan for separating the public CrewPortGlobal website from authenticated functional application navigation.

The target rule is:

```text
Public site = general information + Login / Registration.
Authenticated site = personal cabinet and work areas generated by groups, cards, relationships, authority evidence, tasks and allowed actions.
```

The public website should no longer present all functional areas as public navigation choices.

Functional pages should become authenticated work areas or transitional routes into the personal cabinet.

## 2. Registration, Authentication And Authorization Baseline

The navigation transition must use the approved meaning of registration, authentication and authorization.

### 2.1 Registration

Registration means:

```text
creating or confirming the Physical Person Registration Card / user card
```

Registration answers:

```text
Who is this person?
Which contact data did this person provide?
Which initial path does this person want to start?
```

Registration does not prove account control, company authority, vessel authority, request authority or internal team authority.

### 2.2 Authentication

Authentication means:

```text
confirming access to the Service Account / Login Card
confirming correctness of provided contact data where required
```

Authentication may include:

```text
e-mail verification by one-time code or link
phone verification by SMS, messenger or approved call/message method
future password or SSO verification
future multi-factor verification
```

Authentication answers:

```text
Can this person prove access to the service account?
Can this person prove control of the declared e-mail or phone where required?
```

Authentication does not grant company authority, vessel authority, request authority, broad role access or broad data visibility.

### 2.3 Authorization

Authorization means:

```text
granting powers, roles, data visibility and right to act based on proven evidence and relationship
```

Authorization answers:

```text
Which service areas may this person access?
For which company, vessel, request, client, task or review case may this person act?
Which evidence or relationship proves this right?
Which records and fields may this person see?
Which actions may this person perform?
```

Authorization must be based on group membership, authority evidence, card relationship, task assignment, review/control assignment or Project Owner/admin scope.

## 3. Problem To Correct

The current site has mixed navigation:

```text
public information pages
document pages
functional pages
internal/operator pages
admin/team pages
```

This creates user confusion:

1. public visitors see functional options before registration;
2. seafarers and employer-side clients are pushed into separate public pages instead of a personal cabinet;
3. internal and external concepts appear too close in navigation;
4. direct functional routes can become inconsistent with the new card/access model;
5. future group-based work areas need authenticated menu generation, not static public links.

## 4. Target Navigation Model

CrewPortGlobal should use three navigation layers.

### 4.1 Public navigation

Visible before login.

Purpose:

```text
explain service
build trust
start registration
provide public compliance information
```

### 4.2 Authenticated personal navigation

Visible after account authentication for external users and normal service participants.

Purpose:

```text
show personal cabinet
show own tasks
show cards and workspaces visible by scope
show allowed actions
```

Important:

```text
Authenticated personal navigation may show account and own-card areas.
Business work areas still require authorization by group, evidence, relationship or assignment.
```

### 4.3 Internal/admin navigation

Visible only to authorized team, review, control or Project Owner users.

Purpose:

```text
team work
review/control tasks
admin access
audit
```

## 8. Public Page Content Requirements

Public pages should be shorter, clearer and action-oriented.

Each public page should include:

```text
service summary
who this page is for
what CrewPortGlobal does
what the user must do next
Login / Registration CTA
trust/no-fee/review note where relevant
```

Public pages should not include:

```text
large operational forms
private task lists
operator controls
client records
direct admin/team links
long internal process documentation
```

## 9. Login / Registration Entry Requirements

The public CTA should route to one entry:

```text
/register/
```

Future `/register/` should handle:

```text
create/confirm physical person
create/confirm service account
verify e-mail by code or link where required
verify phone by approved message/call method where required
authenticate service account access
select intended path
create draft path card
route to /me/
```

`/register/` must not:

```text
grant final authority
assign broad role by self-declaration
expose internal team registration
skip duplicate or consent controls
skip required e-mail or phone verification
turn authentication into authorization
```

## 10. Authenticated Navigation Generation

The authenticated menu should be generated from:

```text
GET /api/v1/me/cabinet
```

or an equivalent future endpoint.

The response should include:

```text
visible_sections
menu_items
card_summaries
tasks_summary
allowed_actions
visibility_reasons
presentation_rules
```

Menu item shape:

```text
menu_code
label
href
display_order
visible
badge_count
sla_color
required_permission
visibility_reason
```

The frontend must not hardcode authenticated visibility from role labels alone.

## 11. Shared Navigation Component Requirement

The current shared navigation system should be evolved into two modes.

Recommended modes:

```text
public
authenticated
document
admin
```

Public mode:

```text
renders public menu only
shows Login / Registration
does not show work areas
```

Authenticated mode:

```text
renders menu from cabinet/access response
shows My tasks first where appropriate
shows Logout
```

Document mode:

```text
keeps document navigation for public legal/trust pages
still includes Login / Registration
does not expose functional private links
```

Admin mode:

```text
renders only protected admin navigation after admin session/access check
```

## 12. Direct URL Behavior

Direct URL access must remain safe.

Rules:

1. public route direct access remains public;
2. transitional functional route checks authentication and required contact verification state;
3. unauthenticated user is routed to Login / Registration;
4. authenticated user is routed to the correct `/me/` workspace if visible and authorized;
5. authenticated but unauthorized user receives safe denial or limited cabinet route;
6. internal routes keep their existing protection during transition;
7. admin routes keep admin session/access protection;
8. direct URL must never become the source of permission.

## 13. Public-To-Authenticated User Journey

Recommended journey:

```text
Public home
  -> Login / Registration
  -> physical person registration
  -> service account authentication
  -> e-mail/phone verification where required
  -> path selection
  -> draft card creation
  -> /me/
  -> Мои задачи open
  -> user continues required task
```

Seafarer example:

```text
For seafarers
  -> Login / Registration
  -> I am a seafarer
  -> Seafarer Workforce Card draft
  -> /me/
  -> Мои задачи: complete profile
```

Employer-side example:

```text
For employers
  -> Login / Registration
  -> I request crew / represent employer-side client
  -> Employer-Side Requester Card draft
  -> Company / authority evidence tasks
  -> Crew Request draft
  -> /me/
```

## 14. Menu And Card Presentation Link

Authenticated navigation and card presentation must work together.

Rules:

```text
Мои задачи is the first cabinet card and primary work entry
authenticated menu should also keep My tasks as the first work item
other work areas are available through collapsed cards and menu links
card headers show counts and urgent indicators
menu badges may mirror task/card counts
```

If a section has red/urgent tasks, the menu may show a badge, but details remain scoped.

## 15. Transitional Route UI Requirements

For `/create-profile/` and `/post-vacancy/` during transition:

Unauthenticated users should see:

```text
short explanation
Login / Registration button
no full private form submission
```

Authenticated users should see one of:

```text
redirect/link to /me/seafarer/
redirect/link to /me/employer/
redirect/link to /me/crew-requests/
safe message if access/context is missing
safe prompt to complete e-mail/phone verification when authentication is incomplete
```

These pages must not continue to become parallel private workspaces after the cabinet is introduced.

## 17. Team And Admin Navigation

Team and admin navigation must stay protected.

`/team/`:

```text
requires team or owner group/session as approved
shows links generated by group membership
does not expose secrets
does not appear in public menu
```

`/admin/access/`:

```text
requires admin access session
shows access/admin functions only by permission
does not appear in public menu
audit panel and management functions remain protected
```
