## 1. Purpose

This document converts the BP-003 card model into a practical field dictionary, state model, event model and future database/API requirements.

The goal is to make the next implementation steps concrete:

```text
which cards exist
which fields each card needs
which states each card can have
which events move a card from one state to another
which events generate tasks
which relationships create record visibility
which future database objects and API endpoints will be needed
```

This document remains documentation only. It does not apply migrations or change runtime behavior.

## 2. Design Principles

The card model must preserve these controls:

1. registration, authentication and authorization are separate;
2. a person is not equal to one role;
3. the personal cabinet is assembled from confirmed cards linked to the physical person;
4. employer-side client means buyer / payer / demand side;
5. seafarer means workforce / professional resource / supply side;
6. visibility depends on group membership, authorization evidence and relationship to specific cards;
7. broad group membership must not expose all records;
8. task generation must come from card state, deadline, evidence, relationship or review events;
9. AI may assist with classification and summaries but must not make final legal, employment, payment, reward, compliance or approval decisions;
10. seafarer no-recruitment-fee protection remains mandatory.

## 4. Shared Field Standards

All cards should follow these standards unless a specific exception is approved.

| Field class | Standard |
|---|---|
| Primary ID | UUID generated by backend |
| Foreign key | UUID reference to related card or user |
| State field | Lowercase snake_case enum value |
| Date/time | TIMESTAMPTZ in UTC |
| Money | Decimal amount plus ISO-style currency code |
| Lists | JSONB or normalized child table depending query need |
| Evidence | Stored as reference/metadata, not as public visible raw file by default |
| Notes | Structured internal note plus audit event for material changes |
| Soft removal | Prefer revoked_at / archived_at / closed_at over hard delete |
| Audit | Material create/update/review/visibility events must be auditable |

Shared system fields for all cards:

```text

crew_request_cards

human_review_decisions
card_audit_events
```

Minimum database controls:

1. UUID primary keys;
2. foreign keys where the relationship is stable and known;
3. partial unique indexes for active login e-mail and duplicate-sensitive company identifiers;
4. state check constraints or controlled enum tables;
5. soft-delete / archive fields instead of hard delete for operational records;
6. audit event inserts for material changes;
7. row visibility must be enforced by relationship and scope checks, not by broad group role alone;
8. secrets must not be stored in card tables;
9. raw document files should be stored separately from card metadata;
10. matching summaries must be review inputs, not final employment decisions.

Future API design should provide scoped endpoints for card creation, update, review and visibility.

Candidate endpoint groups:

```text
GET  /api/v1/me/cabinet

GET  /api/v1/employer-requester-cards/{requester_card_id}

```

API rules:

1. `GET /api/v1/me/cabinet` assembles visible cards from authenticated person, group membership and card relationships.
2. List endpoints must be scoped by relationship, assignment, authority evidence or control function.
3. Create endpoints may create drafts but must not grant verified authority automatically.
4. Submit-review endpoints create tasks for the correct group.
5. Human-review endpoints must require authorized human reviewer context.
6. Matching endpoints may generate summaries but must not present candidates without human review.
7. Audit endpoints must be protected and redacted.
8. API responses should expose state, required next action and missing fields.

## 21. Personal Cabinet Requirements

The personal cabinet should assemble:

```text
physical person status
service account status
seafarer workforce card if present
employer-side requester card if present
company contexts visible by authority or relationship
vessel contexts visible by authority or relationship
crew requests visible by authority or relationship
tasks assigned to the user
My clients / relationship-owned cards
internal group pages allowed by group membership
review or control work allowed by group and assignment
```

The personal cabinet must not assemble:

```text
all employer data because the user is an employer-side requester
all seafarer profiles because the user is in a team group
all vessel data because the user is linked to one vessel
all support cases because the user belongs to support_team
```
