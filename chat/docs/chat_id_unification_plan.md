# План перехода на единый `chat_id`

Дата: 26.11.2025  
Автор: GitHub Copilot (GPT-5.1-Codex)

## 1. Цель
Убрать дублирование `session_id`/`chat_id` во фронтенде, PHP-логгере и документации. Единым идентификатором чата становится UUID, который генерирует клиент и отображает пользователю (кнопка «New chat»). Это значение будет передаваться во всех запросах как `chat_id`, а поле `session_id` перестаёт использоваться.

## 2. Актуальное состояние
- **Фронтенд (`chat/index.html`):**
  - Генерирует `session_id` (localStorage `chat:session`), но поддерживает отдельный `chat_id`, который в данный момент может быть пустым.
  - В payload для вебхука присутствуют оба поля, что ведёт к путанице.
- **PHP (`/var/www/html/chat_api.php`):**
  - Требует `session_id` и «при необходимости» создаёт `chat_id`, если он отсутствует.
  - Журнал (`chat_transactions.log`) и SQL-схема работают вокруг `chat_id`, но фактически привязаны к значениям, которые генерирует PHP при отсутствии поля.
- **Документация:** `chat/docs/chat_architecture.md` и `docs/chat_sql_persistence.md` описывают две сущности и процесс создания `chat_id` на сервере.

## 3. План изменений
### 3.1 Фронтенд
1. **Переименование стейта:** `state.sessionId` → `state.chatId`; локальный ключ `chat:session` → `chat:active`. UI элементы, кнопка «New chat» и подсказки должны говорить о `chat_id`.
2. **Генерация идентификатора:** функция `ensureSession()` переименовывается в `ensureChatId()` и продолжает генерировать UUID (fallback `web-<timestamp>`).
3. **Payload:**
   - Во все запросы (`fetch` к вебхуку и `queueSqlLog` → `/chat_api.php`) передаётся только `chat_id` (со значением `state.chatId`).
   - Поле `session_id` удаляется из JSON.
4. **История и UI:** любые подсказки/FAQ о «session_id» обновить до «chat_id». «New chat» = «New chat ID».

### 3.2 PHP (`/var/www/html/chat_api.php`)
1. **Валидация:** требовать `chat_id` как обязательное поле (UUID-строка). Ошибка 400, если отсутствует.
2. **Удалить генерацию:** функция `ensure_chat_id` превращается в `assert_chat_id` (валидация + возврат). Создание чата происходит при первом `mode:log`, используя переданный `chat_id`.
3. **Metadata:** журнал/метаданные больше не добавляют отдельный `session_id`; все события логируют `chat_id`.
4. **Proxy режим:** ожидает уже готовый `chat_id` и не генерирует новый.

### 3.3 Документация
1. `chat/docs/chat_architecture.md`
   - Обновить раздел 5, чтобы payload содержал только `chat_id`.
   - В разделах UI и тест-планов заменить упоминания `session_id` на `chat_id`.
2. `docs/chat_sql_persistence.md`
   - В разделе «Current web chat flow» описать, что `chat_id` генерируется клиентом.
   - В описании API (proxy/log) указать, что `chat_id` обязателен и совпадает с ID, который видит пользователь.
3. Проверить другие документы (FAQ, README) на упоминание `session_id`.

### 3.4 Workflow (n8n)
- Ожидать входящий `chat_id` и возвращать его в ответе.
- Логика построения подсказок/истории фильтрует по `chat_id` + `gtc_user_id`.

### 3.5 Тестирование и проверка
1. **UI:**
   - Запустить чат, убедиться, что отображаемый `chat_id` совпадает с тем, что уходит в payload (можно через DevTools network).
   - Кнопка «New chat» генерирует новый UUID и обнуляет историю.
2. **PHP:**
   - Отправить `mode:log` вручную (curl), убедиться, что запись появляется с переданным `chat_id`.
   - Проверить `chat_transactions.log` — все события должны содержать только `chat_id`.
3. **Workflow:**
   - Убедиться, что ответ содержит `chat_id` и корректно обрабатывается фронтендом.

## 4. Риски и меры
- **Совместимость:** Убедиться, что нигде во внешних сервисах (Stripe, OTP и т.д.) `session_id` не ожидается. Текущий поиск по репозиторию показывает, что бизнес-логика вне чата не зависит от этого поля.
- **Безопасность:** Всегда фильтровать чаты по `gtc_user_id + chat_id`. Пользователь может увидеть ID и поделиться им, но не должен получить чужой чат.

## 5. Готовность к реализации
- n8n конфигурацию владелец настроит отдельно.
- PHP активно использует `session_id`, поэтому правки необходимы (он не «не используется»).
- Документация содержит ~10 упоминаний `session_id`; их нужно обновить в рамках реализации.
