# План интеграции `/auth` во `chat`

## Цель
Объединить существующую страницу `/chat` со всеми возможностями входа/регистрации (`/auth/*`) на одной странице без редиректов пользователя (страница остаётся https://app.gtstor.com/chat/). Сохранить текущую структуру: боковая панель для всей авторизации и служебных кнопок, основная часть для чата.

## План перехода (итерации)
1. **Подготовка**
   - Зафиксировать дизайн auth-блоков (старая страница `gtc_chat.html` служит эталоном по контенту и подсказкам).
   - Добавить в репозиторий чек-лист smoke-тестов (email, регистрация, resend, OTP, Google, подписка) и договориться о QA-окне.
2. **Встраивание форм в `chat/index.html`**
   - Перенести секции «Вход», «Регистрация», «Повторить письмо» в `aside`, используя существующие компоненты `.card`, `.status`.
   - Переиспользовать `postJson` и статусные блоки `setAuthStatus`/`setOtpStatus`, чтобы вся обратная связь отображалась inline.
   - Временно оставить CTA «Перейти на старую форму» (fallback) до завершения тестов — скрыть после успеха.
3. **Единый поток `/auth/*` → `/chat`**
   - После каждой удачной операции вызывать `await fetch('/auth/finish', { credentials: 'include', redirect: 'manual' })`, парсить заголовок `Location` и показывать подсказку (чат/оплата).
   - Немедленно дергать `/auth/profile` (или новый `/auth/status`) и передавать данные в `saveUser()` → `localStorage`.
   - Если `finish` вернул оплату, показывать карточку с кнопкой на `https://pay.gtstor.com/payment.php` как current web billing path и блокировать отправку сообщений. Эта ссылка не описывает Telegram runtime payment flow.
4. **Google и OTP внутри страницы**
   - Интегрировать Google Identity Services (popup/one-tap), отправлять `credential` на `/auth/google`, затем запускать шаг 3.
   - Убедиться, что `/auth/otp/*` вызывает те же статусные обработчики и не требует отдельного UI.
5. **Удаление редиректов**
   - Заменить ссылки `/auth/email`/`/auth/google` на кнопки `type="button"` с JS-обработчиками.
   - Перевести `/auth/index.html` в «минимальный fallback» или редирект на `/chat`.
6. **Отладка и запуск**
   - Обновить `chat/docs/chat_architecture.md`, `docs/Untitled-2` и README.
   - Провести smoke-тесты, собрать метрики (доля пользователей, завершивших вход, сколько переходов на fallback).
   - После недели стабильности выключить старую страницу и удалить её из навигации.

## Шаги реализации (детали)
1. **Разделить нагрузки**
   - Сохранить текущий `aside` в `chat/index.html` как место для управления пользователем: форму email/пароль, OAuth/OTP-кнопки, информационные заметки.
   - Заменить устаревшую ссылку `/auth/google` на компонент, который встраивает обмен `id_token` и защищает ответ.

2. **Взаимодействие с `/auth/*`**
   - Делать POST на `/auth/login`, `/auth/register`, `/auth/request_email_verification`, `/auth/google` из JavaScript формы (аналогично `gtc_chat.html`).
   - После успеха обращаться к `/auth/finish`, но не через изменение `window.location`. Вместо этого:
     1. Делать `fetch('/auth/finish', { method: 'GET', credentials: 'include', redirect: 'manual' })`.
     2. Если сервер возвращает 302 на `/chat` — считать, что user имеет доступ и обновить состояние `state.user` через `saveUser` (возможно, придётся вызвать отдельный эндпоинт `/auth/profile` для получения `gtc_user_id`).
     3. Если возвращается путь оплаты, показать сообщение "У вас нет активной подписки" и, возможно, дать ссылку на оплату.
   - Важно: `auth/finish` очищает `gtc_user_id` cookie после редиректа, поэтому после фетча нужно заново запросить профиль (`/auth/profile`), чтобы получить текущий `gtc_user_id`. Если такой эндпоинт отсутствует, можно либо: (а) расширить `/auth/finish`, чтобы он мог возвращать JSON (совместно с возвратом 200 вместо 302) при запросе `fetch` с `Accept: application/json`, или (б) сделать отдельный `/auth/lookup` на бэкенде.

3. **Защита Google OAuth**
   - Вместо простого `window.location.href` для Google, встроить кнопки, которые инициируют `POST /auth/google` с `google_credential` (как в описании). Это может быть реализовано через интеграцию с `Google Identity Services` на стороне клиента: получить `id_token`, отправить на сервер.
   - После успешного вызова опять же пробежать через `/auth/finish` (см. выше) без изменения URL и обновить UI.

4. **Синхронизация состояния**
   - После каждого успешного обмена обновить `state.user` (email, gtc_user_id) и вызвать `updateAuthUI`. Хранить user в localStorage.
   - Сделать `saveUser` реагировать на новые поля от `/auth/finish` или `/auth/profile`.
   - Возможно, добавить `setTimeout` для проверки подписки и обновления `statusLabel`.

5. **UI и подсказки**
   - Перенести статусные сообщения (`loginStatus`, `signupStatus`) под кнопки и использовать `.status` из UI.
   - Добавить блок FAQ с информацией о `/auth/finish`, `gtc_user_id`, необходимости подписки.
   - Удалить редиректные ссылки, заменив их на вызывные кнопки с `event.preventDefault()`.

6. **Тестирование и откат**
   - Покрыть сценарии: вход, регистрация, повторная отправка письма, Google, отказ в подписке.
   - Оставить возможность быстро сбросить с помощью `resetSession`, `clearHistory` и `logout`.
   - Поддержать работу при отсутствии `gtc_user_id` (при `state.user` null). 

## Зависимости от бэкенда
- Если `/auth/finish` не позволяет запрашивать JSON, предложить добавить параметр `?json=1` или заголовок `Accept: application/json`, чтобы он возвращал `{ status: 'ok'|'denied', redirect: '/chat'|'/payment', gtc_user_id?, email? }` вместо 302.
- Если нельзя менять `/auth/finish`, создать новый маршрут (например, `/auth/status`) который проверяет `AUTH_COOKIE_NAME` и возвращает JSON. Тогда `chat` может обращаться к нему вместо `finish`.
- Убедиться, что `/auth/profile` возвращает нужные поля; если нет, добавить.

## Метрики успеха
- Пользователь остаётся на `/chat`, не видит никаких редиректов.
- После входа/регистрации/Google/верификации появляется `gtc_user_id` и кнопки `New chat/Clear chat` становятся доступными.
- При отсутствии подписки показывается понятное сообщение с инструкцией оплатить.
